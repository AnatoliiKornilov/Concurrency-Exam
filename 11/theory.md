# 11. Системные вызовы select, poll и epoll. Разница между ними. Реализация многопоточного сервера через epoll.

## Проблема: как обрабатывать тысячи соединений без busy loop?

В busy loop polling мы постоянно опрашивали все сокеты, тратя CPU. Блокирующий I/O на поток на клиент не масштабируется. Нужен механизм, который позволяет **усыпить поток** до тех пор, пока на **любом** из наблюдаемых дескрипторов не произойдёт событие (данные для чтения, возможность записи, ошибка). Пробуждение происходит только при наличии события.

## Две модели: `reactor` и `proactor`
+ `Reactor` (модель готовности, readiness): ядро уведомляет «дескриптор готов к операции», а саму операцию (`read/write`) выполняете вы. Так работают `select`, `poll`, `epoll`. На операцию приходится минимум два сисколла: «дождаться готовности» + «сделать I/O».
+ `Proactor` (модель завершения, completion): вы просите ядро выполнить операцию, и оно уведомляет «операция завершена, вот результат/данные». Так работает `io_uring` (Linux). Сисколл «сделать I/O» исчезает — он совмещён с уведомлением.`

## `select()`

### Цель `select()`
`select()` – это системный вызов, который позволяет одному потоку наблюдать за несколькими файловыми дескрипторами (сокетами, файлами, pipe и т.д.) и заблокироваться до тех пор, пока хотя бы один из этих дескрипторов не станет «готов» к операции ввода-вывода (чтение, запись, или не возникнет исключительная ситуация).

После того как `select()` вернулся, поток узнаёт, какие именно дескрипторы готовы, и может безопасно выполнить на них `read()` или `write()`, зная, что операция не заблокируется.

Ключевая идея: не нужно крутить `busy loop` и постоянно опрашивать дескрипторы. `select()` усыпляет поток до наступления события.

### Сигнатура и параметры

```c
int select(int nfds, fd_set* readfds, fd_set* writefds, fd_set* exceptfds, struct timeval *timeout);
```

+ `nfds` – максимальный номер дескриптора среди всех переданных плюс один. Ядру нужно знать, до какого номера сканировать битовые маски (оптимизация). Например, если вы передаёте дескрипторы 5, 17 и 42, то nfds должно быть 43 (максимальный номер + 1).

+ `readfds` – множество дескрипторов, которые мы хотим проверить на готовность к чтению (данные уже есть в буфере сокета, можно читать без блокировки).

+ `writefds` – множество дескрипторов, которые мы хотим проверить на готовность к записи (в буфере отправки есть место, можно писать без блокировки).

+ `exceptfds` – множество для исключительных ситуаций (обычно используются для OOB-данных в TCP, редко).

+ `timeout` – максимальное время ожидания. Если `NULL` – ждать бесконечно. Если указано время – ждать не дольше, затем вернуться даже если событий не было (вернёт 0).

### Что делает `select()` под капотом

1. Проверка аргументов – ядро проверяет, что переданные дескрипторы корректны.

2. Копирование множеств – ядро копирует битовые маски из пользовательского пространства в своё внутреннее представление.

3. Опрос (`polling`) в ядре – ядро проходит по всем дескрипторам от `0` до `nfds-1`, проверяя для каждого, в каком множестве он присутствует, и запрашивает у драйвера устройства (например, сетевого стека) состояние дескриптора. Если дескриптор не готов, ядро запоминает, что поток хочет ждать этот дескриптор.

4. Засыпание – если ни один дескриптор из запрошенных ещё не готов, ядро переводит поток в состояние сна (`TASK_INTERRUPTIBLE`) и добавляет его в очереди ожидания для каждого дескриптора.

5. Пробуждение – когда на каком-то дескрипторе происходит событие (например, приходят данные в сокет), драйвер вызывает функцию пробуждения, которая удаляет поток из очередей ожидания и помечает его как готовый к выполнению. Планировщик позже запустит поток.

6. Заполнение множеств – перед возвратом в пользовательское пространство, ядро модифицирует переданные множества `readfds`, `writefds`, `exceptfds`, оставляя в них только те дескрипторы, которые действительно готовы. То есть исходные множества теряются.

7. Возврат – `select()` возвращает количество готовых дескрипторов (сумма по всем трём множествам), или 0, если сработал таймаут, или -1 при ошибке.

### Проблемы `select()`

+ Ограничение `FD_SETSIZE` – обычно 1024. Нельзя наблюдать за дескриптором с номером ≥ 1024. Можно перекомпилировать libc с большим значением, но это не переносимо.

+ Линейное сканирование – ядро проходит по всем дескрипторам от `0` до `nfds-1` при каждом вызове, даже если активных мало. При большом `max_fd` (например, `100000`) это очень медленно.

+ Копирование масок – каждая маска (до 1024 бит) копируется туда и обратно. Это не так уж дорого для 1024, но для 100000 – нельзя.

+ Сложность `O(N)` в ядре и `O(N)` в пользовательском коде для проверки всех дескрипторов.

+ Модификация множеств – нужно каждый раз пересоздавать копию перед вызовом.

## `poll()`

### Цель `poll()`

`poll()` делает то же самое, что и `select()`: позволяет потоку наблюдать за множеством файловых дескрипторов и блокироваться, пока хотя бы один из них не станет готов к вводу-выводу. Но `poll()` решает некоторые ограничения `select()` и имеет более удобный интерфейс.

### Сигнатура и параметры

```c
int poll(struct pollfd* fds, nfds_t nfds, int timeout);
```

+ `fds` – указатель на массив структур `struct pollfd`. Каждая структура описывает один дескриптор и интересующие нас события.

+ `nfds` – количество элементов в массиве (тип `nfds_t` – беззнаковое целое).

+ `timeout` – время ожидания в миллисекундах. Отрицательное = бесконечно, 0 = не ждать (проверить и вернуться сразу).

Структура struct `pollfd`:

```c
struct pollfd {
  int   fd;         // дескриптор (если fd = -1, запись игнорируется)
  short events;     // битовая маска событий, которые мы хотим отслеживать (входной параметр)
  short revents;    // битовая маска событий, которые реально произошли (заполняется ядром)
};
```

Основные флаги `events` / `revents`:

+ `POLLIN` -	Данные для чтения (обычный read не заблокируется)
+ `POLLOUT` -	Можно записывать (буфер отправки не заполнен)
+ `POLLERR` -	Ошибка (обычно возвращается в revents как признак ошибки)
+ `POLLHUP` -	Соединение разорвано (hang up)
+ `POLLNVAL` -	Неверный запрос (fd закрыт или не открыт)

### Что делает `poll()` под капотом

1. Проверка массива – ядро проверяет каждый элемент массива `fds`. Если fd == -1, элемент пропускается.

2. Копирование массива – ядро копирует массив `fds` из пользовательского пространства в своё (один раз на вызов).

3. Опрос (`polling`) – ядро проходит по всем элементам массива и для каждого дескриптора вызывает функцию ядра, проверяющую его состояние (через драйвер устройства). Если дескриптор уже готов (есть данные, можно писать), ядро устанавливает соответствующие биты в `revents` и помечает, что вызов может вернуться немедленно.

4. Засыпание – если ни один дескриптор не готов, ядро переводит поток в состояние сна и добавляет его в очереди ожидания для каждого из дескрипторов.

5. Пробуждение – когда на каком-то дескрипторе происходит событие, драйвер пробуждает поток, ядро удаляет его из всех очередей ожидания и обновляет `revents` для готовых дескрипторов.

6. Копирование обратно – ядро копирует массив `fds` обратно в пользовательское пространство, чтобы пользовательский код мог прочитать `revents`.

7. Возврат – `poll()` возвращает количество элементов, у которых `revents` не ноль (количество готовых дескрипторов), или 0 по таймауту, или -1 при ошибке.

+ Важно: `poll()` не изменяет `events` и не удаляет записи. Он только заполняет `revents`. Поэтому массив `fds` можно использовать повторно, не пересобирая его полностью (в отличие от `select`).

### Плюсы и минусы `poll()`

#### Плюсы:

+ Ключевое преимущество `poll`: нет верхнего предела номера дескриптора. Вы можете работать с любыми дескрипторами (например, 100000). Но всё ещё есть линейная проверка всех дескрипторов при каждом вызове (`nfds`). Если у вас 10000 дескрипторов, но активны 5, `poll` всё равно пробежится по 10000 записям в ядре каждый раз.

#### Минусы:

+ Линейное сканирование – ядро проверяет все записи в массиве при каждом вызове. При большом количестве дескрипторов (сотни тысяч) это очень дорого.

+ Копирование массива – при каждом вызове `poll()` весь массив копируется из userspace в kernel и обратно. Для большого массива (например, 100000 элементов) это значительные накладные расходы.

+ Сложность `O(nfds)` – не зависит от того, сколько дескрипторов активно.

## `epoll()`

### Цель `epoll()`

`epoll` – это механизм мультиплексирования ввода-вывода, специфичный для Linux. Он решает главные проблемы `select` и `poll`:

+ Линейное сканирование – в `select`/`poll` ядро при каждом вызове проверяет все переданные дескрипторы. При десятках тысяч соединений это очень медленно.

+ Копирование данных – каждый вызов копирует битовые маски (`select`) или массив структур (`poll`) из пользовательского пространства в ядро и обратно.

+ Ограничение на количество дескрипторов – `epoll` не имеет жёсткого предела, ограничено только памятью ядра.

+ `epoll` работает асинхронно/событийно: ядро уведомляет приложение только о тех дескрипторах, на которых реально произошли события, и делает это за `O(1)` (или `O(активные дескрипторы)`, а не `O(все зарегистрированные)`).

### Сигнатура и параметры

`epoll` состоит из трёх функций:

```c
#include <sys/epoll.h>

int epoll_create1(int flags);
int epoll_ctl(int epfd, int op, int fd, struct epoll_event* event);
int epoll_wait(int epfd, struct epoll_event* events, int maxevents, int timeout);
```

1. `epoll_create1(flags)`

+ Создаёт "экземпляр `epoll`" в ядре и возвращает файловый дескриптор, который будет использоваться для всех последующих операций.

+ `flags` обычно 0 (можно `EPOLL_CLOEXEC` – автоматически закрывать при `exec`).

+ Возвращает `epoll`-дескриптор (например, `epfd`).

2. `epoll_ctl(epfd, op, fd, event)`

Добавляет, удаляет или модифицирует интересующий дескриптор в epoll-экземпляре.

+ `op`:

+ `EPOLL_CTL_ADD` – добавить новый дескриптор.

+ `EPOLL_CTL_MOD` – изменить события для существующего дескриптора.

+ `EPOLL_CTL_DEL` – удалить дескриптор (больше не следить).

+ `fd` – целевой дескриптор (сокет, файл, pipe).

+ `event` – указатель на структуру `struct epoll_event`, описывающую интересующие события и пользовательские данные.

```c
struct epoll_event {
  uint32_t events;   // битовая маска событий (EPOLLIN, EPOLLOUT, EPOLLET...)
  union {
    void* ptr;
    int fd;
    uint32_t u32;
    uint64_t u64;
  } data;            // пользовательские данные (обычно кладут fd или указатель на структуру)
};
```

3. `epoll_wait(epfd, events, maxevents, timeout)`

+ Ожидает события на дескрипторах, зарегистрированных в `epoll`. Блокирует поток до наступления события или таймаута.

+ `events` – массив структур `epoll_event`, куда ядро запишет события.

+ `maxevents` – размер этого массива (максимум сколько событий за раз можно получить).

+ `timeout` – время ожидания в миллисекундах; -1 – бесконечно; 0 – немедленный возврат (опрос).

Возвращает количество готовых дескрипторов (записанных в `events`), 0 по таймауту, -1 при ошибке.

### Как работает `epoll` под капотом
1. В ядре для каждого `epoll`-экземпляра поддерживаются две основные структуры:

+ Красно-чёрное дерево – хранит все дескрипторы, зарегистрированные через `epoll_ctl`. Поиск, вставка, удаление за `O(log N)`.

+ Готовый список (`ready list`) – связный список дескрипторов, на которых произошло событие.

2. Регистрация дескриптора (`epoll_ctl ADD`): Ядро добавляет узел в дерево и устанавливает обратный вызов (`callback`) в драйвере устройства (например, в сетевом стеке), который будет вызываться при наступлении события. Этот `callback` просто добавляет дескриптор в готовый список (если его там ещё нет) и будит поток(и), ожидающие на `epoll_wait`.

3. Вызов `epoll_wait`: Если готовый список не пуст – ядро копирует события из списка в пользовательский массив и очищает список (или удаляет только переданные, зависит от режима).
Если список пуст – ядро усыпляет вызывающий поток и добавляет его в очередь ожидания этого `epoll`-экземпляра.
При появлении события `callback` добавит дескриптор в готовый список и разбудит поток(и).

4. Удаление дескриптора (`epoll_ctl DEL`): Удаляет узел из дерева, отменяет `callback`.

### Режимы работы `epoll`: `Level-Triggered (LT)` и `Edge-Triggered (ET)`
+ `Level-Triggered (LT)` – поведение по умолчанию

Если на дескрипторе есть данные (т.е. `read` не заблокируется), `epoll_wait` будет продолжать возвращать этот дескриптор в каждом вызове, пока вы не прочитаете все данные.
Аналогично `select/poll`.
Проще в использовании, менее подвержен ошибкам.

+ `Edge-Triggered (ET)` – более эффективный режим

Событие генерируется только один раз, когда состояние дескриптора меняется с «нет данных» на «есть данные».
Приложение должно читать до тех пор, пока read не вернёт `EAGAIN`, иначе можно потерять данные.
Требует неблокирующих сокетов и более аккуратного кода.
Позволяет значительно снизить количество пробуждений (особенно при большом трафике).
Включение `ET: ev.events = EPOLLIN | EPOLLET;`

+ Улучшение для многопоточности: часто используют `EPOLLONESHOT`, чтобы после события дескриптор автоматически удалялся из `epoll` до тех пор, пока приложение снова его не добавит. Это позволяет безопасно передавать обработку в другой поток без риска повторных событий.
