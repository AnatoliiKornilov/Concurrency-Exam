# 16. Когерентность кэшей. Протокол MESI. Cache ping-pong. Проблема false sharing и ее решение.

## Почему кэш так важен?

Процессоры работают намного быстрее, чем память (RAM). Чтобы скрыть эту разницу, используются **кэши** — небольшая, но очень быстрая память, расположенная близко к ядру. Типичная иерархия: L1 (32–64 КБ на ядро), L2 (256–512 КБ), L3 (несколько МБ, общий для всех ядер). Кэш хранит копии данных из оперативной памяти.

## Проблема когерентности кэшей

Когда несколько ядер имеют свои кэши, они могут работать с одними и теми же данными. Если одно ядро изменяет значение в своём кэше, другие ядра не видят это изменение — возникает **несогласованность кэшей** (cache incoherency). Нужен аппаратный механизм, который обеспечивает **когерентность** – все ядра видят одинаковое значение по одному адресу.

## Протокол MESI

MESI — самый распространённый протокол когерентности кэшей (Intel, AMD, ARM). Каждая **кэш-линия** (обычно 64 байта) находится в одном из четырёх состояний:

| Состояние | Описание |
|-----------|----------|
| **M** (Modified) | Линия только в этом кэше, изменена (грязная). Данные в памяти устарели. Запись в эту линию не требует оповещения других кэшей. |
| **E** (Exclusive) | Линия только в этом кэше, но согласована с памятью (не изменена). Запись переведёт её в M. |
| **S** (Shared) | Линия может быть в нескольких кэшах, все копии согласованы с памятью. Запись потребует оповещения всех. |
| **I** (Invalid) | Линия недействительна (кэш не содержит актуальных данных). |

### Переходы состояний (упрощённо)

- **Чтение**:
  - Если линия в M/E/S → читаем без оповещения.
  - Если I → посылаем запрос на шину. Если никто не имеет (E/M) → становится E. Если кто-то имеет в S → становится S.
- **Запись**:
  - Если линия в M → просто пишем.
  - Если в E → переходим в M.
  - Если в S → посылаем "RFO" (Read For Ownership) — инвалидируем копии в других кэшах, переходим в M.
  - Если в I → сначала RFO, затем запись.
- **Другие ядра**:
  - Если слышат чтение от другого ядра, то при наличии M -> сброс в память и переход в S.
  - Если слышат RFO, то инвалидируют свою копию (I).

## Cache ping-pong (кэш-пинг-понг)

Возникает, когда несколько ядер постоянно пишут в одну и ту же кэш-линию (или соседние переменные, попавшие в одну линию). Каждая запись требует RFO, что заставляет другие кэши инвалидировать свою копию и перезагружать её при следующем чтении. Потоки гоняют кэш-линию между ядрами — резкое падение производительности.

## False sharing (ложное разделение)

Это частный случай cache ping-pong, когда **разные потоки пишут в разные переменные, но эти переменные находятся на одной кэш-линии**. Ядру «кажется», что потоки конкурируют за одни данные, и оно перекладывает линию между кэшами, хотя фактически конфликта нет.

### Пример false sharing на C++

```cpp
struct Data {
    int x;   // поток 1 пишет только в x
    int y;   // поток 2 пишет только в y
};
// если x и y на одной кэш-линии (обычно 64 байта), то записи в x инвалидируют кэш-линию у потока 2, и наоборот.
```

Решение: выравнивание (padding) — увеличить расстояние между переменными так, чтобы они гарантированно лежали в разных кэш-линиях.

```cpp
struct alignas(64) Data {
    int x;
    char padding[60]; // чтобы x занимал всю линию
    int y;
};
// или с помощью атрибутов компилятора:
struct Data {
    int x;
    int y;
} __attribute__((aligned(64)));
```

С `C++17` можно использовать `std::hardware_destructive_interference_size` и `std::hardware_constructive_interference_size` для переносимого размера кэш-линии.
