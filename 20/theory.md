# 1. Проблема управления памятью в lock-free структурах

В lock-free алгоритмах (например, стек Трайбера, очередь Майкла-Скотта) потоки работают с атомарными указателями и циклами CAS. При удалении узла (pop) возникает опасная ситуация:

- Поток A прочитал указатель на узел X (например, head).
- Поток B успевает удалить узел X (delete) и создать новый узел, который может быть размещён по тому же адресу.
- Поток A продолжает работу, использует старый адрес (уже освобождённый или переиспользованный) – **Use-After-Free** и **ABA problem**.

**Суть проблемы:** нельзя просто вызвать `delete`, пока какой-либо другой поток может держать ссылку на этот узел. Нужен механизм, который гарантирует, что память освобождается только тогда, когда на неё нет активных ссылок из других потоков.

## Методы решения (Safe Memory Reclamation, SMR)

Существует несколько подходов. Мы рассмотрим два основных:

## Hazard Pointers (Указатели опасности)

**Идея:** каждый поток перед тем, как получить доступ к узлу (прочитать его указатель, начать CAS), объявляет этот указатель как **«опасный»** (hazard pointer) — публикует в специальном массиве, видимом для всех потоков. Другой поток, желающий удалить узел, проверяет все опасные указатели всех потоков. Если указатель на узел находится в любом hazard pointer, удаление откладывается. Если нет – можно удалять безопасно.

### Как работает

1. **Поток инициализирует** свой массив hazard pointers (обычно фиксированный размер, например 5, на поток).
2. **Перед чтением** (например, `head.load()`):
   - Поток записывает адрес узла в свой hazard pointer (атомарно).
   - Повторно проверяет, что указатель не изменился (иначе повторяет).
3. **После использования** (когда больше не нужен) – затирает hazard pointer (например, `nullptr`).
4. **При удалении** (например, в `pop` после успешного CAS):
   - Поток, желающий удалить узел, собирает все hazard pointers всех потоков.
   - Если указатель удаляемого узла есть в каком-то hazard pointer – откладывает удаление (например, добавляет в локальный список «для удаления позже»).
   - Если нет – может безопасно выполнить `delete`.

### Преимущества и недостатки

- **Плюсы:** относительно простая реализация, хорошо подходит для структур, где число активных ссылок невелико.
- **Минусы:** накладные расходы на поддержание hazard pointers (каждый доступ требует записи в массив), может тормозить при высокой конкуренции. Не защищает от ABA полностью (решает только use-after-free, ABA можно решать отдельно счётчиками).

### Упрощённый пример (идея)

```cpp
thread_local HazardPointer hp; // массив указателей для текущего потока

Node* pop() {
    while (true) {
        Node* h = head.load();
        if (h == nullptr) return nullptr;
        hp.set(0, h); // объявляем опасный указатель
        if (head.load() != h) continue; // защита от изменения
        Node* next = h->next.load();
        if (head.compare_exchange_strong(h, next)) {
            // успешно извлекли
            // проверяем, не используется ли h кем-то как hazard pointer
            if (!is_hazard(h)) {
                delete h;
            } else {
                retire(h); // отложить удаление
            }
            return h->data;
        }
    }
}
```

## Read-Copy-Update (RCU)
+ Идея: RCU изначально разработан для ядра Linux, но применим и в пользовательском пространстве. Позволяет читателям получать доступ к данным без блокировок (lock-free reads), а писатели создают новые версии данных. Удаление старых данных откладывается до тех пор, пока все читатели, которые могли захватить указатель на старые данные, не завершат свои критические секции (читательские секции RCU).

### Ключевые понятия RCU
+ Читательская секция – участок кода, в котором поток обращается к защищённым данным. Вход в секцию помечается `rcu_read_lock()`, выход – `rcu_read_unlock()`. Внутри секции поток не должен спать (блокироваться) и не должен менять данные.

+ Грация (`grace period`) – интервал времени, в течение которого все потоки, находившиеся в читательских секциях на момент начала периода, успели их покинуть. После завершения `grace period` можно безопасно освобождать память старых версий.

### Как работает RCU
1. Чтение:

+ Поток вызывает `rcu_read_lock()`, который обычно просто увеличивает локальный счётчик (или помечает поток как читатель).

+ Поток читает указатель на данные (атомарно).

+ Поток вызывает `rcu_read_unlock()`.

2. Обновление (удаление):

+ Писатель создаёт новую версию данных (например, новый узел).

+ Атомарно заменяет глобальный указатель на новую версию.

+ Теперь старая версия больше не доступна для новых читателей.

+ Писатель вызывает `synchronize_rcu()`, который ждёт завершения всех читательских секций, которые начались до замены (т.е. ждёт, пока все потоки выйдут из своих секций). Это и есть `grace period`.

+ После возврата из `synchronize_rcu()` писатель может безопасно удалить старые данные.

### Преимущества и недостатки RCU
+ **Плюсы**: очень высокая производительность для read-интенсивных сценариев; нет накладных расходов на hazard pointers на каждую операцию.

+ **Минусы**: сложность реализации; требует, чтобы читательские секции были короткими и не содержали блокировок; `grace period` может быть долгим, увеличивая задержку освобождения памяти.

## Класс `std::atomic_shared_ptr`

###  Отличие от обычного `shared_ptr`

Обычный `std::shared_ptr` не является потокобезопасным для операций записи. Несколько потоков не могут одновременно вызывать `store`, `reset` или `swap` на одном объекте `shared_ptr` без внешней синхронизации. Однако разные `shared_ptr`, указывающие на один объект, безопасны для чтения (копирование, увеличение счётчика ссылок) — это исключение для счётчика ссылок (он атомарен).

`std::atomic_shared_ptr` – это специализация `std::atomic<T>` для `T = std::shared_ptr`. Он предоставляет все атомарные операции: `load`, `store`, `exchange`, `compare_exchange_weak/strong`, `wait`, `notify_one` и т.д. Это позволяет безопасно менять умный указатель в многопоточной среде без дополнительных мьютексов.

## Идея реализации
Реализация `std::atomic_shared_ptr` должна решить задачу атомарного изменения двух указателей: указателя на объект и указателя на блок управления (control block). Есть несколько подходов:

1. С использованием двойного CAS (Double-Word CAS)
На платформах, поддерживающих `cmpxchg16b` (x86-64), можно упаковать два 64-битных указателя в 128-битную структуру и выполнять над ней атомарные операции.

Тогда std::`atomic_shared_ptr` может быть lock-free. Однако такой подход не портабелен (ARM, RISC-V могут не иметь 128-битного CAS).

b) Использование мьютекса внутри (blocking)
Самый простой способ – хранить внутри `std::mutex` и выполнять операции под блокировкой. Такой `atomic_shared_ptr` будет блокирующим (не lock-free), но зато переносимым и безопасным. Именно так часто реализовано в стандартных библиотеках (GCC libstdc++ использует мьютекс для больших типов, но для `shared_ptr` может быть своя оптимизация).

c) Использование техники «указатель + флаг блокировки»
Можно выделить один бит в указателе на блок управления и использовать его как `spinlock`. Тогда `atomic_shared_ptr` будет реализован через атомарный спинлок – это уже ближе к lock-free (но всё равно требует ожидания). Однако такой подход сложен и не гарантирует прогресса для всех операций.

d) Использование `Hazard Pointers` или `RCU` для управления блоком управления
Экзотический вариант, но теоретически возможен. На практике `std::atomic_shared_ptr` обычно либо блокирующий, либо lock-free только на некоторых платформах.

## Можно ли сделать lock-free `atomic_shared_ptr`?
Да, можно, но это требует:

+ Аппаратной поддержки двойного CAS (например, double_compare_exchange).

+ Или очень изощрённого использования свободных битов в указателе (например, на 64-битных системах адреса обычно 48-битные, можно хранить счётчик в старших битах). Это позволяет уместить и указатель на данные, и указатель на блок управления, и счётчик в 128 бит или меньше.

+ Однако стандарт C++ не требует lock-free реализации для `std::atomic_shared_ptr`. Поэтому разработчики библиотек часто выбирают простой и надёжный путь – блокирующую реализацию (мьютекс).
