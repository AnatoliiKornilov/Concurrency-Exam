# 15. Файберы. Идея реализации context switch. Scheduler, функция yield. Fault injection.

## 1. Файбер как модель потока

Прежде чем строить планировщик, полезно понять, **зачем** файберы вообще и почему, изучив их, мы лучше понимаем **потоки**. Игрушечный однопоточный кооперативный файбер — простая, но честная **модель потока**: разобравшись в его механике, вы получаете хорошее приближение к тому, как устроены и потоки, которые вы запускаете в пуле.

### 1.1. Механика потока (напоминание)

Ядро исполняет инструкции, читая их из памяти; у него есть `instruction pointer` (на очередную инструкцию) и `stack pointer` (на стек, по которому ядро прыгает в функции и возвращается). Поток живёт на ядре, пока не уйдёт **сам** (системный вызов — добровольная уступка) или не будет **вытеснен недобровольно** по прерыванию от системного таймера: прерывание переводит процессор в привилегированный режим, исполняется код планировщика ОС, и тот решает, что запускать дальше. Файбер моделирует ровно это, только переключение всегда **добровольное** (см. `Yield`, §4.3).

### 1.2. Поток vs файбер: в чём разница

- **Кооперативность vs вытеснение.** Наши файберы **не вытесняются**: если файбер не отдаёт управление, он работает, пока не завершится или не вызовет `Yield`. Потоки ОС вытесняются недобровольно.
- **Одно ядро vs параллельность.** Эти файберы исполняются в одном потоке (том, где крутится планировщик); потоки ОС бегут физически параллельно на разных ядрах. Файберы **станут** параллельными — нужен лишь многопоточный планировщик (§9).
- **Имена.** thread = «нить», fiber = «волокно»: нить сплетается из волокон. В ограниченном числе потоков (пул) исполняется много волокон-файберов.
- **И то, и другое — в user space.** Потоки `pthread` тоже библиотечные: `pthread` сам аллоцирует стек, реализует свои примитивы синхронизации. Единственное реальное отличие — `pthread` зовёт `clone` (сажает исполнение на отдельное ядро ОС), а файбер — нет; за «посадку на ядро» у нас отвечает **пул потоков**, а не файбер (и это большое преимущество дизайна, §9). Аллокация стеков и примитивы синхронизации у файберов — по сути эквивалентный потокам код.

### 1.3. Почему файберы кооперативны (и это нормально)

Вытеснение файберам не нужно **не потому, что мы не можем**, а потому, что **не требуется** для их задачи. Файберы обслуживают **IO-bound**-код: сотни тысяч и миллионы клиентов в микросервисе/распределённой системе (хоть бы и простой эхо-сервер) не **считают**, а спят на операциях ввода-вывода. Эти 100k–1M файберов будут «параллельно спать», постоянно обращаясь к IO, мьютексам, каналам, — а точки ввода-вывода и есть естественные точки **кооперативного** переключения. Файбер просто не задерживается на ядре надолго и не претендует на процессор, так что вытеснять его незачем. Код, который **претендует** на процессор (тяжёлые вычисления), на файберах писать не надо — файбер не обязанность, а инструмент для IO-bound-сценариев. Для сравнения: Go вытесняет горутины, потому что в Go **весь** код — горутина (можно написать счётный цикл, не обращающийся к рантайму, и его придётся прервать); у нас же выбор за вами. На практике невытесняемые файберы прекрасно обслуживают эксабайты данных (например, в YDB).

### 1.4. Три применения файберов в курсе

1. **Модель потока** — понять механику потоков (эта лекция).
2. **Средство выражения concurrency** — обслуживать много клиентов масштабируемо (главная цель; §9 и файлы про корутины/IO).
3. **Детерминированная симуляция и тестирование** — управляемая инъекция переключений ловит гонки, deadlock, livelock (§8).

## 2. Повторение: контексты

В предыдущем файле реализованы низкоуровневые `SwitchContext` и `SetupContext` (ассемблер), сохраняющие/восстанавливающие callee-saved регистры и `RSP` и переключающие стеки. Кратко:

- `SwitchContext(void** from_rsp, void* to_rsp)` — сохраняет текущий `RSP` в `*from_rsp`, грузит новый из `to_rsp`, восстанавливает регистры, делает `ret`, переходя на новый стек.
- `SetupContext(stack_bottom, trampoline, arg)` — готовит пустой стек так, чтобы после первого `SwitchContext` управление перешло в `StaticTrampoline`, вызывающую `arg->Run()`.

Цель — построить на этих примитивах полноценную систему файберов: планировщик, ожидание (suspend/resume), таймеры, многопоточное исполнение.

## 3. Архитектура однопоточного файбера

### 3.1. Класс `Fiber`

```cpp
class Fiber : public IntrusiveNode<Fiber>, public ITrampoline, public IAwaitable {
public:
    enum class State { Runnable, Running, Dead, Suspended };

    template <class F>
    explicit Fiber(F&& f, Handle* handle)
        : routine_(std::forward<F>(f)), handle_(handle) {
        ctx_.Setup(stack_.GetStackView(), this);   // через SetupContext; this как ITrampoline
    }

    void Run() final;          // точка входа (из трамплина)
private:
    Routine routine_;          // пользовательская функция
    Stack stack_;              // собственный стек (обычно 4–8 КБ; см. про guard page и пул в файле про context switch)
    Context ctx_;              // контекст (сохранённый RSP)
    Handle* handle_;           // связанный handle (join/detach)
    Fiber* waiter_;            // файбер, ожидающий нас (для синхронизации)
    State state_ = State::Runnable;
    std::span<std::byte> tls_view_;   // аналог thread-local storage
};
```

**Ключевое:** наследование от `ITrampoline` позволяет передать `this` в `SetupContext` — после первого переключения вызовется `Fiber::Run()`. Стек выделяется классом `Stack` (выровнен; о guard page и пуле стеков — в файле про context switch). `waiter_` — ожидающий завершения файбер (для `join`).

### 3.2. Запуск: `Fiber::Run()`

```cpp
void Fiber::Run() {
    state_ = State::Running;
    try {
        alignas(16) std::array<std::byte, kStorageSize> tls{};  // "TLS" прямо на стеке файбера
        tls_view_ = tls;
        routine_();             // пользовательский код
    } catch (...) {
        // подавляем все исключения, чтобы не пробить дно стека (раскрутка за пределы памяти = UB)
    }
    state_ = State::Dead;
    if (handle_) handle_->detach();
    Scheduler::This().Schedule(this);   // возврат в планировщик
    UNREACHABLE();              // сюда не возвращаемся
}
```

Локальный `tls` на стеке файбера служит аналогом `thread_local`. После завершения `routine_` состояние → `Dead`, файбер отдаёт управление планировщику; `Schedule` для мёртвого файбера немедленно переключается в планировщик (`SwitchToSched`), затем файбер удаляется.

### 3.3. Деструктор и пробуждение ожидающих

```cpp
Fiber::~Fiber() {
    if (waiter_) {
        waiter_->state_ = Fiber::State::Runnable;
        Scheduler::This().Schedule(waiter_);
    }
}
```

Если кто-то сделал `join()` на этом файбере, в деструкторе пробуждаем ожидающего и ставим в очередь.

## 4. Планировщик (`Scheduler`)

В однопоточном варианте — `thread_local` объект:

```cpp
static thread_local Scheduler scheduler;
Scheduler& Scheduler::This() { return scheduler; }

class Scheduler {
public:
    void Run();                          // главный цикл
    void Schedule(Fiber* fiber);         // добавить в очередь
    void Suspend(IAwaitable* awaitable); // уснуть на ожидании
    void Sleep(Timer* timer);            // уснуть на таймере
    void Yield();                        // добровольно уступить
    static Scheduler& This();
    static Fiber* Running();
private:
    FiberQueue fibers_;                  // очередь готовых
    TimerQueue timers_;                  // очередь таймеров
    Context sched_ctx_;                  // контекст планировщика
    Fiber* running_ = nullptr;
};
```

### 4.1. Очереди

- `FiberQueue` — **интрузивный двусвязный список** (`Fiber` наследует `IntrusiveNode<Fiber>`): вставка/удаление без аллокаций, указатели `next`/`prev` внутри объекта.
- `TimerQueue` — упорядочена по времени срабатывания (например, `priority_queue`).

### 4.2. Главный цикл и шаг

```cpp
void Scheduler::Run() {
    while (Step()) { /* одна итерация: файбер или таймеры */ }
}

bool Scheduler::FiberStep() {
    auto fiber = fibers_.PickRandom(/*runnable=*/true);
    if (!fiber) return false;
    fibers_.Remove(fiber);
    running_ = fiber;
    running_->SetState(Fiber::State::Running);
    SwitchToFiber(running_);             // sched_ctx_.Switch(fiber->GetContext())
    running_ = nullptr;
    if (fiber->GetState() == Fiber::State::Dead) delete fiber;
    else if (fiber->GetState() == Fiber::State::Runnable) fibers_.Prepend(fiber);
    // Suspended — файбер уже в чьей-то очереди ожидания (например, таймера)
    return true;
}
```

`Step()` сначала пробует запустить готовый файбер (`FiberStep`), затем обрабатывает таймеры (`TimerPoll`); если ни тех, ни других нет — цикл завершается. `sched_ctx_` — контекст планировщика, сохраняется при входе в файбер и восстанавливается, когда файбер вызовет `SwitchToSched()`.

### 4.3. Возврат в планировщик и `Yield`

```cpp
void Scheduler::SwitchToSched() {
    if (running_->GetState() == Fiber::State::Dead)
        running_->GetContext().SwitchLast(sched_ctx_);   // больше не вернёмся (корректно для санитайзеров)
    else
        running_->GetContext().Switch(sched_ctx_);
}

void Scheduler::Yield() {
    ASSERT(running_->GetState() == Fiber::State::Running);
    running_->SetState(Fiber::State::Runnable);
    SwitchToSched();
}
```

`Yield` помечает файбер `Runnable` и возвращает управление планировщику; тот может снова запустить его или выбрать другой. Это ключ к **fault injection** (§8).

### 4.4. `Yield` как диспетчеризация системного вызова

`Yield` — «странная функция»: вы входите в один вызов, а выходите из **другого** (управление окажется в другом файбере или в планировщике). Механически это происходит так: файбер находит планировщик (глобальная/`thread_local` ссылка) и переключается на его контекст (`SwitchToSched`), но при этом передаёт планировщику **колбэк-обработчик**, который исполнится уже **на стеке планировщика** и решит, что делать с остановившимся файбером. Для `Yield` обработчик кладёт файбер в конец `run queue`; для `Suspend` (§5) — паркует на объекте ожидания; для завершения — помечает `Dead`. То есть `Yield`/`Suspend` — это, по сути, **диспетчеризация системного вызова**: файбер «зовёт ядро» (планировщик), а конкретное действие выбирается обработчиком на стороне планировщика (почему файбер вернул управление, знает он сам, но он уже не активен — поэтому намерение и передаётся колбэком). База для всего этого — что `sched_ctx_` всегда валиден: мы **попали** в файбер из планировщика через `Switch`, так что вернуться всегда есть куда (контекст самого `main`-потока изначально создан ОС).

### 4.5. Цена: `yield` ломает структурное программирование

У этой мощи есть оборотная сторона. Когда-то человечество отказалось от `goto` в пользу **структурного программирования** (Дейкстра): вызовы образуют аккуратную систему вложенных/непересекающихся отрезков, и об исполнении легко рассуждать. `yield` это ломает: вход в один вызов, выход из другого — структура исполнения превращается в «лапшу». Чтобы промоделировать программу в уме, приходится держать в голове `run queue` планировщика — шаг назад и по качеству программ, и по когнитивной нагрузке (а ещё это мешает решать ряд промышленных задач). Эту «лапшу» однажды и «чинят» **stackless-корутины**, где компилятор восстанавливает структуру, генерируя автомат (см. файл про stackless-корутины, единая трансформация «цикл/рекурсия → автомат»).

## 5. Ожидание и примитивы синхронизации

### 5.1. `IAwaitable` и `Suspend`

```cpp
class IAwaitable {
public:
    virtual void Park(Fiber* fiber) = 0;   // запарковать файбер на этом объекте
};

void Scheduler::Suspend(IAwaitable* awaitable) {
    running_->SetState(Fiber::State::Suspended);
    awaitable->Park(running_);             // делегируем парковку объекту ожидания
    SwitchToSched();                       // уходим в планировщик
    // когда нас разбудят — окажемся здесь и продолжим
}
```

Любой объект, на котором файбер может уснуть (таймер, мьютекс, условная переменная, другой файбер), реализует `Park`. Пробуждающий код выставляет `state = Runnable` и вызывает `Scheduler::Schedule(fiber)`.

### 5.2. Таймер (`Sleep`)

```cpp
void Scheduler::Sleep(Timer* timer) { timers_.Add(timer); Suspend(timer); }

bool Scheduler::TimerPoll() {
    if (timers_.Empty()) return false;
    Timepoint tp = Now();
    while (!timers_.Empty() && timers_.Top()->CompareWithTimepoint(tp)) {
        auto fiber = timers_.Top()->Unpark();
        timers_.Pop();
        fiber->SetState(Fiber::State::Runnable);
        Schedule(fiber);
    }
    return true;
}
```

`Timer` наследует `IAwaitable`; его `Park` запоминает файбер и время пробуждения, а `TimerPoll` при наступлении времени возвращает файбер в очередь готовых.

## 6. `Handle`: управление временем жизни

`Handle` — аналог `std::thread` для файберов (`join`/`detach`).

```cpp
class Handle {
public:
    template <class F>
    explicit Handle(F&& f) : fiber_(new Fiber(std::forward<F>(f), this)) { Schedule(); }
    void detach();
    void join();
    bool joinable();
    ~Handle() { ASSERT(!joinable()); }
private:
    void Schedule() { Scheduler::This().Schedule(fiber_); }
    Fiber* fiber_{};
};
```

- При создании `Handle` новый `Fiber` сразу попадает в очередь планировщика.
- `join()` — текущий файбер делает `Scheduler::Suspend(fiber_)`, паркуясь на целевом; завершившийся целевой в деструкторе пробуждает ожидающего.
- `detach()` обнуляет связь — файбер продолжает независимо.
- `Handle` некопируем, только перемещаем.

## 7. Интрузивные списки (`IntrusiveNode`)

```cpp
template <typename T>
class IntrusiveNode { T* next_{nullptr}; T* prev_{nullptr}; /* вставка/удаление */ };
```

Преимущества: нет динамических аллокаций при добавлении в очередь (указатели внутри объекта), экономия памяти и лучшая локальность. `FiberQueue` — обёртка над интрузивным списком (`Prepend`, `Remove`, `PickRandom`).

## 8. Тестирование и fault injection

Однопоточные файберы идеальны для **детерминированного тестирования** многопоточного кода:

- Код на `std::atomic`/`std::mutex` компилируется в режиме файберов (без потоков ОС).
- Планировщик **внедряет дополнительные `Yield()`** в критические точки (например, внутри методов обёрток над атомиками), провоцируя гонки (race conditions), которые на реальных потоках всплывают редко.
- Исполнение становится полностью **детерминированным и воспроизводимым** — упавший тест воспроизводится по seed'у

Конфигурации сборки (проект `lines`):
- **`LINES_THREADS`** — настоящие потоки ОС и `std::atomic`.
- **Без `LINES_THREADS`** — файберы и обёртки над атомиками с инъекцией `Yield()`.

Сама инъекция тривиальна: «атомик» в режиме файберов — это обёртка, которая перед каждой операцией добровольно уступает планировщику, а тот в тесте умеет вставлять `Yield()` в «неудобный» момент:

```cpp
template <class T>
struct FaultyAtomic {           // подменяет std::atomic<T> в режиме файберов
    T value;
    T load() const            { InjectFault(); return value; }
    void store(T v)           { InjectFault(); value = v; }
    bool compare_exchange_strong(T& exp, T des) {
        InjectFault();                       // переключение ровно между чтением и записью CAS
        if (value == exp) { value = des; return true; }
        exp = value; return false;
    }
};
void InjectFault() {            // точка управляемого переключения
    if (ShouldPreempt())        // детерминированно по seed'у (или с заданной вероятностью)
        Fiber::Yield();         // отдать управление — спровоцировать межпоточное чередование
}
```

Поскольку всё крутится в одном потоке, `Yield()` в `InjectFault` детерминированно переставляет файберы местами в критических точках, и редкие гонки воспроизводятся по seed'у.

Санитайзеры:
- **ASAN** работает с файберами, если при переключении звать `__sanitizer_start_switch_fiber` / `__sanitizer_finish_switch_fiber` (иначе ASAN путается в границах стеков).
- **TSAN** — через `__tsan_switch_to_fiber`; для однопоточных файберов он не нужен (гонок нет), для многопоточных помогает ловить data race.

## 9. Многопоточные файберы и корутины

### 9.1. Проблема масштабирования

Однопоточный планировщик ограничен одним ядром. Для масштабирования нужны несколько рабочих потоков, распределение файберов между ними и синхронизация общих структур (очередей). Прямое добавление мьютексов в `Scheduler` усложняет код и бьёт по производительности. Альтернатива — **корутины + executor**.

### 9.2. Stackful-корутина

В лекции — простая stackful-корутина, похожая на файбер, но с **симметричной передачей управления без планировщика**:
- `Resume()` — запустить/продолжить;
- `Suspend()` — приостановить и вернуть управление вызвавшему;
- `IsCompleted()` — завершена ли.

Внутри использует тот же `Context`/`SetupContext`, но управление передаётся явно между вызывающим и корутиной (без очереди планировщика). Тесты: `JustWorks` (последовательные Suspend/Resume), `Stackful` (`Suspend` из любой вложенной функции — отличие от stackless, см. файл про корутины), `Folded` (вложенные корутины).

### 9.3. Файберы поверх корутин + executor

Идея:
- Заменить `Fiber` на `Coroutine`.
- Вместо планировщика с очередью — **executor** (например, thread pool), исполняющий задачи.
- Когда файбер хочет уступить (`yield`) или уснуть (`sleep`), он не переключается в планировщик, а **разрывает** корутину, сохраняя контекст, и планирует её продолжение как отдельную задачу в executor.

```cpp
void Fiber::Suspend(IAwaitable* awaitable) {
    awaitable->Park(this);
    auto continuation = [this]() { this->state_ = State::Running; this->Resume(); };
    executor_->Submit(std::move(continuation));   // продолжение — отдельная задача
    Coroutine::Suspend();                          // немедленно вернуть управление текущему потоку
}
```

Каждый «кусок» файбера между точками ожидания становится задачей, исполнимой на любом потоке. Синхронизация упрощается: общие структуры (очереди таймеров) защищаются обычными мьютексами, а планирование берёт на себя executor.

**Связь с пройденным.** Это, по сути, модель **M:N** (много файберов на немного потоков ОС), как у горутин Go (планировщик G-M-P). На стороне executor'а естественно ложится **work-stealing** (см. файл про пул потоков и futures): у каждого рабочего потока своя дека задач-продолжений, простаивающий ворует у соседа — так распределяется нагрузка без contention на единой очереди. Executor'ы здесь — тот же примитив, что в файле про futures: они задают, **где** исполняется продолжение. Подробный разбор эффективного work-stealing планировщика (G-M-P, локальные/глобальная очереди, воровство, двухфазная парковка, FIFO/LIFO) — в приложении «Work-Stealing планировщик».

### 9.4. `go`/`yield`: файбер как цепочка задач

Самое сложное в многопоточных файберах — не синхронизация, а **декомпозиция** на два ортогональных компонента: **корутина** (см. файл про stackless-корутины — обобщение подпрограммы, умеет `resume`/`suspend`, ничего не знает про планировщик и параллелизм) и **планировщик = пул потоков** (умеет исполнять задачи, ничего не знает про корутины и файберы). Сложив их, получаем `go`/`yield` буквально в несколько строк:

```cpp
// go: запустить файбер = завернуть процедуру в корутину и сабмитить служебную задачу с resume
void Go(IExecutor& pool, Routine f) {
    auto* co = new Coroutine(std::move(f));
    pool.Submit([co] { co->Resume(); });        // первый шаг файбера
}

// yield: остановить файбер и запланировать его продолжение отдельной задачей
void Yield() {
    Coroutine* co = CurrentCoroutine();
    pool.Submit([co] { co->Resume(); });        // следующий шаг — будущая задача
    co->Suspend();                               // вернуть управление воркеру пула
}
```

Отсюда определение со стороны рантайма: **файбер = цепочка задач**. Каждая задача делает `co->Resume()`; дойдя до `Yield`, корутина останавливается (`Suspend` возвращает управление воркеру = коллеру), но прежде планирует **новую** задачу на следующий `Resume`. Воркер пула, завершив одну задачу, берёт следующую — возможно, служебную задачу другого файбера. Так файберы чередуются и параллельно исполняются на потоках пула.

Ценность именно в **ортогональности**: пул не знает про файберы (исполняет любые задачи, в т. ч. служебные задачи файберов — см. файл про condition variable/thread pool), корутина не знает про многопоточность. Сложность **факторизуется**: можно сделать пул умнее (work-stealing, §9.3) — `go`/`yield` не изменятся ни на строку, потому что опираются лишь на гарантию `Submit`: «задача однажды выполнится». В коде `go`/`yield` нет ни переключения контекста, ни потоков — только `resume`/`submit`; вся низкоуровневая сложность спрятана под двумя понятиями.

### 9.5. Синхронизация файберов: останавливаем файбер, а не поток

Зафиксировав число потоков (пул), мы запретили блокирующие вызовы: заблокированный поток крадёт у программы виртуальное ядро (гигиена из файла про пул). Но синхронизация и ввод-вывод по природе блокирующи — спасает та же корутина. **Fiber-mutex** либо захватывается сразу, либо **останавливает файбер** (его корутину): `Suspend` паркует файбер в очередь ожидания мьютекса и возвращает поток пулу; при `unlock` ожидающий файбер планируется новой задачей и однажды `Resume`-ится. Поток при этом **не блокируется** ни на миг. Поэтому для файберов не нужен futex — нужен «**futex для файберов**» (приостановка файбера вместо парковки потока ядром). На том же принципе строятся каналы и `select` — вся синхронизация файберов делается **без** блокировки потоков пула. Ввод-вывод решается аналогично, но требует более развитого рантайма, чем простой пул (event loop) — см. файл про async/callback/context и про IO-мультиплексирование.

> В пределе эту картину можно перевернуть: сделать **future** центральным примитивом, а синхронизацию (event), таймеры и I/O выразить как futures, которые файберы/потоки/корутины ждут одним и тем же `await`. Тогда сам файбер становится **деталью реализации** «future-first» рантайма (служебные carrier-файберы, на которых исполняются задачи), а у пользователя остаётся единый API. Подробно — в приложении «A2. Lazy Futures».

## 10. Итог

- Файбер — простая **модель потока**: кооперативный, в user space, от `pthread` отличается по сути лишь отсутствием `clone` (посадку на ядро берёт пул); кооперативность достаточна, т. к. файберы IO-bound (спят, а не считают). Три применения: модель потока / средство concurrency / детерминированное тестирование.
- Файбер = `SwitchContext`/`SetupContext` (механизм) + `Scheduler` (политика) + интеграция с `epoll`.
- `Scheduler`: интрузивная очередь готовых, таймеры, `Yield`/`Suspend`/`Sleep`, `Handle` для `join`/`detach`. `Yield`/`Suspend` — диспетчеризация «системного вызова» к планировщику; ценой служит сломанное структурное программирование (его восстанавливают stackless-корутины).
- Управляемая инъекция `Yield()` делает однопоточные файберы инструментом **детерминированного тестирования** гонок (fault injection).
- Масштабирование на много ядер — ортогональная декомпозиция **файбер = корутина + пул потоков**: `go` = корутина + `submit(resume)`, `yield` = `suspend` + `submit(resume)`, файбер = **цепочка задач**. Сложность факторизуется. Модель **M:N** с work-stealing; синхронизация файберов **останавливает файбер, а не поток** («futex для файберов», каналы, `select`).
