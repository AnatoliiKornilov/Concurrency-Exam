# 14. Реализация классов std::future и std::promise. В чем их неудобство?

## Устройство. 
`Future` и `promise` делят shared state — объект в куче, содержащий:

+ `std::mutex` для синхронизации;

+ `std::condition_variable` для блокировки ожидающих;

+ `std::variant<std::monostate, T, std::exception_ptr>` (значение или исключение) 
+ флаг готовности.
`get()` под мьютексом ждёт на condition `variable`, пока не выставлен флаг готовности, затем возвращает значение или перебрасывает исключение. 
`wait_for(0)` просто проверяет флаг.

## Неудобства `std::future`:

+ Только блокирующие `wait/get`. Нельзя «подписаться» на готовность, чтобы выполнить код без блокировки потока.

+ Требует динамической аллокации `shared state`.

+ Нет удобной композиции: ни `then` (`continuation`), ни комбинаторов.

+ Нельзя задать, в каком потоке выполнять продолжение.

+ Деструктор `future` от `std::async` блокирует (`launch::async`). Это известная ловушка: временный объект `std::async(f);` без присваивания исполняется фактически синхронно, потому что его деструктор ждёт завершения задачи. Аналогично цепочка `std::async(...).get()` не даёт параллелизма, если `future` немедленно разрушается.

+ Один результат — один потребитель. Нескольким ждущим нужен `std::shared_future` (его можно копировать, и `get()` зовут многократно из разных потоков).

Чего хочется: `future.Subscribe(callback)` — `callback` выполнится, когда результат будет готов. Это открывает путь к асинхронным цепочкам без блокировки.
