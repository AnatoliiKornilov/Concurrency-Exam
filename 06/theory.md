## Data race (гонка данных)

**Гонка данных** возникает, когда:
- два или более потока одновременно обращаются к одной области памяти,
- хотя бы один из них выполняет **запись**,
- и эти обращения не упорядочены (нет синхронизации).

В C++ гонка данных приводит к **UB**.

### Пример

```cpp
int counter = 0;          // общая переменная
// Поток 1: counter++;
// Поток 2: counter++;
```

Операция `counter++` неатомарна (читает, модифицирует, записывает). Два потока могут прочитать 0, оба увеличить до 1 и записать – результат 1, а должен быть 2.

## Store buffering и переупорядочивание
Современные процессоры и компиляторы могут переупорядочивать операции для оптимизации. Это может привести к неожиданным эффектам, даже при использовании атомарных операций без барьеров.

## Проблема Store Buffering
Рассмотрим два потока, работающих с двумя переменными:

Поток 1	Поток 2

`x = 1`	`y = 1`

`r1 = y`	`r2 = x`

На архитектурах с ослабленной моделью памяти возможно, что оба потока увидят r1 = 0 и r2 = 0, если записи задержались в буферах store, а чтения выполнились раньше. Это противоречит интуиции, но реально.

Решение – использовать барьеры памяти или атомарные операции с определёнными `memory_order`(например, `memory_order_seq_cst`).

## Класс `std::atomic<T>`
Обеспечивает атомарный доступ к значению типа ``T. Для тривиальных типов обычно реализован без блокировок (`lock-free`).

### Основные методы:

+ `load()` - атомарно прочитать значение
+ `store(val)` - атомарно записать значение
+ `exchange(val)` -	атомарно записать и вернуть предыдущее значение
+ `compare_exchange_weak(expected, desired)` -	если `текущее == expected`, то записать `desired`; иначе записать текущее в `expected`. Может ложно срабатывать (`spurious failure`) – нужен цикл.
+ `compare_exchange_strong` -	как `weak`, но без ложных отказов
+ `fetch_add(val), fetch_sub, fetch_and, fetch_or, fetch_xor` -	атомарно выполнить операцию и вернуть старое значение
+ `is_lock_free()` -	проверить, использует ли атомик блокировки (обычно для больших типов)

## Spinlock (спинлок)
+ Спинлок – примитив синхронизации, при котором поток вместо засыпания выполняет активное ожидание (проверку флага в цикле).

### Когда полезен:
+ Критическая секция очень короткая (несколько инструкций)

+ Конкуренция низкая, ожидание редко

+ В системах с реальным временем, где недопустимо переключение контекста

+ В ядре ОС

### Когда вреден
+ Долгое ожидание (тратит процессор впустую)

+ Высокая конкуренция (увеличивает задержки)

+ На однопроцессорных системах (если поток, удерживающий спинлок, вытеснен, остальные будут ждать вечно)

### `Mutex` и `Spinlock`
+ Вместо `Mutex` можно использовать `Spinlock`, так как у них одинаковое апи, однако надо понимать,
что у каждой из этих струкутру свои плюсы и минусы.
